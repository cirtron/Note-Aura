# 更新日誌

Note-Aura 所有重要變更均記錄於此，最新在前。

## 2026-06-27

### 新增
- **管理員：IP 位址封鎖。** 管理員現在可從全新的 **管理 → 封鎖的 IP 位址** 頁封鎖 IP 位址。被封鎖的 IP 會在中介層解析工作階段之前以 HTTP 403 拒絕。封鎖清單支援選填的原因說明，並顯示每個 IP 被封鎖的時間。解除封鎖即時生效。
- **管理員：可設定的登入鎖定。** 連續登入失敗達設定次數後，帳號會鎖定指定的分鐘數。兩個閾值均在 **管理 → 登入鎖定** 中設定（`login_max_attempts`，預設 5；`login_lockout_minutes`，預設 15）。成功登入後計數器重置。管理員可在使用者頁面解除任何使用者的鎖定。
- **管理員：強制登出與定時停權。** 使用者頁面新增 **強制登出** 按鈕，可立即讓某使用者所有進行中的工作階段失效；停權表單也支援輸入選填的小時數進行定時停權（留空 = 永久）。定時停權將在指定時間自動解除。
- **網站公告橫幅。** 管理員可在 **管理 → 網站公告** 撰寫全站公告並切換顯示。啟用後，訊息會以琥珀色橫幅顯示在每個頁面頂部（導覽列下方）。
- **管理員：發送電郵。** 管理員可透過 **管理 → 發送電郵** 發送純文字電郵給指定使用者、某角色的所有使用者，或某群組的所有成員。需要設定 SMTP；未設定時頁面會顯示提示。
- **登入鎖定錯誤訊息。** 登入頁現在會針對以下情況顯示具體且已翻譯的錯誤訊息：憑證無效、帳號鎖定（含剩餘分鐘數）、帳號停權、email 未驗證 —— 不再只顯示單一的「電郵或密碼無效」。四種訊息均已翻譯成英文、繁體中文、简体中文與日本語。

## 2026-06-26

### 新增
- **筆記瀏覽頁顯示附件。** 上傳圖片或檔案後，筆記頁面現在會在內文底部顯示專屬的 **附件** 區塊：
  - **每個檔案**（圖片、`.txt`、`.docx`、`.pdf`、影片等）均有 `📎 檔名（大小）` 下載連結。
  - **圖片** 還會在下載連結下方顯示內嵌預覽。
  - 此區塊以 `attachments` 資料庫資料表為依據，即使筆記內文只有 OCR 擷取的文字而 Markdown 中未嵌入圖片 URL 也同樣有效。
- **筆記表單的一次性提交按鈕。** 新增筆記 / 編輯筆記頁面上所有提交按鈕（URL 擷取、圖片上傳、檔案上傳、手動筆記表單）在第一次點擊後立即停用，防止在伺服器重新導向前因雙擊或網路緩慢而建立重複筆記。

### 修正
- **驗證碼 Cookie 重放攻擊防護。** 圖片驗證碼驗證 cookie 現在在每次驗證後（無論通過或失敗）立即過期，因此同一個挑戰 token 無法被重複使用。下次頁面渲染時會如常發放新的 cookie 與圖片。
- **重新寄送驗證信表單缺少驗證碼檢查。** 登入頁上的「重新寄送驗證信」連結在寄送新驗證信之前沒有驗證驗證碼，與登入 / 註冊 / 忘記密碼表單不一致。現在統一先驗證驗證碼，與其他驗證流程一致。
- **OCR 與圖片分析現在以使用者選擇的語言輸出。** 從圖片擷取的文字（OCR）與圖片描述輸出現在使用使用者選擇的 **摘要語言** 生成（例如繁體中文 / 简体中文）。這與現有的標題、摘要、分類生成行為一致。Ollama 與雲端（OpenAI 相容）提供者均已更新；語言指示附加在送出圖片的提示詞中。
- **OCR 產生重複內容（相同文字出現兩次，以 `[Image] ---` 分隔）。** 當 OCR 已回傳 ≥ 50 個 rune 的文字時，現在會跳過圖片描述（`Describe`）呼叫。對於文字密集的文件圖片，OCR 與 Describe 使用同一個視覺模型，會重新讀取文字而非提供視覺背景，造成帶有 `[Image]` 前綴的逐字重複。對於 OCR 回傳空白或很少文字的照片與圖表，Describe 仍照常執行。
- **`update.ps1`：伺服器輸出被靜默丟棄。** `Start-Process` 啟動執行檔時未重新導向 stdout/stderr，因此啟動錯誤不可見。腳本現在透過 `cmd /c ... >> note-aura.log 2>&1` 啟動（與 `update.sh` 的 `nohup >>` 行為一致），將所有輸出追加到專案資料夾的 `note-aura.log`。

## 2026-06-21

### 新增
- **登入 / 註冊圖片驗證碼。** 登入、註冊、忘記密碼表單現在顯示扭曲的 **圖片驗證碼**（5 個英數字元）；你需要輸入圖片中顯示的字元。此功能可阻止自動化註冊與密碼猜測攻擊，為內建功能，不需要外部服務（Cloudflare/Google）也不需要 API 金鑰，因此適用於離線/企業內網部署。圖片在伺服器生成，驗證碼只存放在短期有效的 HMAC 簽署 cookie 中（不以明文傳送到瀏覽器）；輸入錯誤時一定會換一張新圖片。輸入不區分大小寫。標籤已翻譯成英文、繁體中文、简体中文與日本語。
- **行動版篩選器。** 在手機上，筆記清單現在有一個可折疊的 **🔍 篩選** 面板，直接位於選單下方，提供先前只在較大螢幕上可見的分類（「專案」）與標籤篩選器。此面板重用現有的篩選資料，並突顯目前使用中的分類/標籤；桌面版側邊欄不受影響。

## 2026-06-17

### 新增
- **Facebook 連結擷取。** 貼上 Facebook 連結（貼文、影片、Reels 或 Watch 頁面），Note-Aura 會像處理一般網頁或 YouTube 連結一樣擷取其內容 —— 擷取標題、文字與任何可用的字幕，然後 AI 生成標題、分類、標籤與摘要。來源標籤顯示 `facebook`。會依序嘗試三種擷取方式：yt-dlp 含 cookies（最豐富 —— 影片中繼資料 + 字幕）、已驗證 headless Chrome（文字貼文）、公開 Open-Graph 預覽（備援，不需要 cookies）。管理員可透過 **管理 → AI 設定 → Facebook cookies (Netscape cookies.txt)** 設定 cookies 路徑；留白時仍可取得大多數連結的公開預覽。
- **AI 處理的停止按鈕。** 筆記在 `processing` 狀態時，擁有者和編輯者會在狀態標籤旁看到 **停止** 按鈕。點擊後會取消進行中的 AI 工作、刪除排隊中的工作，並將筆記轉換至中性的「已停止」狀態。筆記瀏覽頁會顯示中性框（「AI 處理已停止。」）與靛藍色 **重試** 按鈕，取代原本的紅色錯誤框。重試時會清除已停止旗標並重新排隊 AI。所有 UI 字串均已翻譯成英文、繁體中文、简体中文與日本語。

### 新增
- **Worker 生命週期日誌。** Worker 現在在開始工作時記錄 `processing note N job M (source=…)`，完成時記錄 `completed note N job M`（之前只在失敗時記錄）。這使得我們能夠看出緩慢的 OCR 是否真的在執行，還是根本沒被撿起來。
- **啟動時記錄有效的 AI 設定。** Note-Aura 啟動時現在記錄 `AI config: ollama_url=… vision_model=… per-call timeout=…`，這樣 OCR/逾時問題可以從日誌診斷 —— 例如確認在重新部署後調高 `ai.timeout_seconds` 是否真的生效。
- **`update.sh` —— Linux/macOS 更新工具。** `update.ps1` 的對應版本：在伺服器上從原始碼目錄執行，可停止 → 重新編譯 → 重新啟動，同時保留所有資料。自動偵測 systemd（重新啟動 `note-aura` 服務，可用 `NOTE_AURA_SERVICE` 覆寫）或在背景重新啟動。已在 README 與 INSTALL.md §9 中記錄。

### 修正
- **OCR 逾時（「context deadline exceeded」）且逾時設定被忽略。** 兩個 AI 提供者都硬式編碼了 180 秒的 HTTP 客戶端逾時，因此調高 config.yaml 中的 `ai.timeout_seconds` 對 Ollama/雲端呼叫沒有效果。由於生成是非串流的，那個單一逾時必須涵蓋*整個*回應 —— 而在 CPU 或遠端主機上冷啟動 `deepseek-ocr` 並推論的時間通常超過 180 秒，導致每次 OCR 失敗。提供者的 HTTP 逾時現在由 `ai.timeout_seconds` 控制，預設值已提高至 **600 秒**，且 config 說明已解釋此設定。如果你的視覺主機非常慢，可進一步調高（例如 900）。
- **YouTube 字幕經常缺失（只擷取到描述）。** 字幕擷取使用 `yt-dlp … -o -`，但 yt-dlp **不會**將字幕輸出到 stdout —— 它會寫入一個字面上叫做 `-.<lang>.vtt` 的檔案，因此主要擷取總是回傳空白（並在工作目錄留下垃圾檔案），導致每部影片都退回到容易受速率限制的備援。yt-dlp 現在將字幕軌下載到臨時目錄並讀取 `.vtt` 檔案，並加入 `--retries`/`--retry-sleep` 來緩解 YouTube 的 HTTP 429 速率限制。結果：有字幕的影片現在能可靠地擷取字幕；臨時目錄自動移除（不再有多餘的 `-.en.vtt`）。
- **`deepseek-ocr` 的 OCR / 圖片分析沒有產生任何輸出。** 預設視覺模型 `deepseek-ocr` 是一個針對特定提示詞訓練的專家型、對換行字元敏感的模型；Note-Aura 傳送了它未被訓練的通用指示（「從這張圖片擷取所有文字…」），導致空白/亂碼輸出。預設的 OCR 與圖片分析提示詞現在使用模型說明文件中的格式（`Free OCR.` / `Describe this image in detail.`，以及必要的前導換行符），與有效的 OmniScribe 參考一致。圖片分析失敗原本被靜默吞掉，現在會記錄到管理員系統日誌。**注意：** 如果你在 管理 → AI 設定 下設定了自訂 OCR/圖片提示詞，請清除它（以取得新預設值）或將其設為 `Free OCR.`；deepseek-ocr 還需要 Ollama ≥ v0.13.0。
- **YouTube 反機器人頁面被擷取為字幕。** 當 YouTube/Google 對伺服器 IP 進行速率限制時，字幕端點回傳了「We're sorry…」HTML 頁面，但沒有任何檢查確認它是真正的字幕 —— 因此該文字被清理後存為影片字幕。字幕內容現在只有在帶有 `WEBVTT` 簽名時才被接受（新的 `parseCaption` 防護，同時應用於 yt-dlp stdout 路徑和中繼資料字幕 URL 路徑；後者現在直接擷取軌道原始內容，而非透過防機器人 HTML 擷取器）。被封鎖的擷取會退回影片描述，而不是污染筆記。
- **已停止筆記的復活競爭（worker）。** 如果某個 `provider.*` AI 呼叫恰好在 `Cancel()` 與下一次 DB 寫入之間的窄視窗內完成，該寫入會覆寫停止處理器的 `status=failed, stopped=1` 狀態，使筆記留在不一致的 `status=ready, stopped=1` 狀態（顯示為正常的就緒筆記）。現在在 `run()` 中每個可能覆寫已停止/失敗狀態的持久化寫入之前立即套用取消防護（`ctx.Err()` 檢查）：`ApplyAIResult` 呼叫（無 AI 路徑、早期儲存、摘要後）、`SetNoteTags`、`SetNoteCategory` 和 `ReplaceChunks`。已取消的 context 現在會在這些寫入之前回傳錯誤，讓 `process` 走現有的無操作路徑，並保留已停止狀態。
- **完整多語介面。** 每個頁面現在均以 English / 繁體中文 / 简体中文 / 日本語 渲染，包括先前只有英文的密碼重設、忘記密碼、問筆記、行事曆等頁面。`TestLanguageKeyParity` 測試強制確保每個翻譯鍵在所有四種語言中都存在。
- **個人語言記憶。** 使用者選擇的介面語言儲存在其帳號中，並在各裝置間跟隨（登入前以 cookie 記憶）。
- **本地化的交易性電郵。** 驗證和密碼重設電郵以收件者的語言寄送；邀請電郵以寄件者在邀請表單上選擇的語言寄送（在「重新寄送」時重複使用）。
